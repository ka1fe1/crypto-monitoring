# Code Review 输出与追踪规则（始终生效）

> 目的：将评审结论从聊天输出固化为可追踪文档，驱动 coding agent 按照反馈进行修复与二次验证，形成闭环。

## 默认落盘规则（覆盖 chat-only 习惯）

- 触发条件：用户提出“review/代码评审/评审当前变更/对最新变更进行 review”等诉求时，默认视为**已明确要求生成评审文档**。
- 默认输出：
  - 必须在 `/docs/review/` 下创建当日序号文档 `YYYYMMDD-NNN.md`，并按本文档结构完整填充。
  - chat 中只输出：总体结论（APPROVE / REQUEST\_CHANGES / COMMENT）+ 文档链接（file:/// 形式）+ 关键问题概览（可选）。
- 例外条件：仅当用户明确说“不要落文档/只在 chat 输出/不要写 docs/review”时，才允许不创建评审文档。

## 适用范围

- 日常缺陷修复、功能迭代以及专项治理所有变更的代码评审
- 集成测试相关代码除外

## 输出位置与命名

- 文档路径：`/docs/review/`
- 命名规范：`YYYYMMDD-NNN.md`
  - `YYYYMMDD` 为评审日期（本地时区）
  - `NNN` 为当日评审序号，从 `001` 递增（例：同一天第二份评审为 `YYYYMMDD-002.md`）
- 新问题创建规则：
  - 在“验证修复”的过程中如果发现**新的问题**，新建下一号评审文档（例如 `YYYYMMDD-002.md`），**不要**在旧文档中追加，以避免序号错乱与维护成本上升
  - **同问题复检失败不新建**：若问题属于旧文档已登记项的“未完全修复/修复方式不满足原建议/同链路回归”，应在原文档追踪表更新
      `review 结果=未通过` 与“未通过原因”，不新建文档
  - **仅独立新问题才新建**：只有当问题与旧文档问题集无直接关联（新增模块/新增缺陷类型/新增风险面）时，才新建下一号文档
  - **用户指定验证某文档时优先回填**：用户明确要求“验证 `YYYYMMDD-NNN.md` 修复情况”时，默认先在该文档完成结论回填；若出现独立新问题，再按规则新增下一号文档

## 文档结构（必须）

1. 元信息（Meta）
   - 评审日期、评审人、分支/Commit 范围、执行的测试/构建命令、总体结论（APPROVE / REQUEST\_CHANGES / COMMENT）
2. 变更范围摘要
   - 涉及文件与模块，按功能域归类（Job/Webhook/Repository/Adapter/Entity/Spec 等）
3. 发现清单（Findings）
   - 按严重级别分组：P0 / P1 / P2 / P3
   - 每个问题包含：简述、影响/场景、定位（文件+行号范围）、修复建议（具体到接口/条件/状态机）
   - 每个分组下列表的序号必须从 1 开始，依次递增
4. 补充说明（可选）
   - 设计权衡、边界条件、风险与回归点
5. 问题处理追踪表（必须，见文末模板）
   - 用于记录后续修复与二次评审结果

## 严重级别定义（统一）

- P0 Critical：安全/丢数/确定性错误，必须阻断合并
- P1 High：明显逻辑/并发/一致性风险，建议本轮修复
- P2 Medium：可维护性/语义一致性/性能可预见风险，可随版本迭代
- P3 Low：工程卫生/命名/日志/小型重构，非阻断

## 编写要求（必须）

- 代码定位：使用项目相对路径 + 行号范围链接，示例：
  - `[job.go:L570-L612](../../internal/service/job/job.go#L570-L612)`
- 事实为先：给出复现实验或命令（测试/构建/集成）与结论
- 建议可执行：修复建议须明确到“位置/条件/状态转移/SQL 过滤/幂等点

## 定位链接维护策略（行号不要求长期同步）

- 定位链接（文件 + 行号范围）是**评审当时的快照**，主要用于帮助 coding agent 在当轮/当次修复中快速定位问题。
- 规则要求：在创建评审文档时，必须提供可跳转的定位链接（相对路径 + 行号范围）。
- 不要求：在后续代码演进（无论是同 PR 的额外提交还是后续版本迭代）中，**不强制维护**历史评审文档里的行号准确性，也不要求因为行号漂移去更新旧文档链接。
- 二次评审：review agent 只验证“问题是否已按建议修复、测试是否通过、追踪表是否更新”，不以“旧行号仍然精确命中”作为通过标准。
- 新问题：验证修复过程中发现新的问题，按规则新建下一号评审文档记录，不在旧文档追加或为旧问题持续追踪行号变化。
  - 例外：若属于旧问题复检未通过或旧问题衍生回归，继续在原文档追踪表回填，不新建文档。

## 通过/未通过判定与后续动作

- 审核通过（APPROVE）：
  - coding agent 在修复后更新表格“修复结果/补充描述”
  - review agent 二次验证后更新“review 结果/未通过原因”并标记为“通过”，该问题视为最终关闭
- 审核未通过（REQUEST\_CHANGES/COMMENT）：
  - coding agent 必须根据“未通过原因”重新修复，并更新表格中的“修复结果/补充描述”
  - review agent 二次验证后更新“review 结果/未通过原因”，若仍未通过则需继续迭代
- 发现新问题：新建 `YYYYMMDD-002.md` 等新文档记录，不在旧文档中新增问题
  - 若为旧问题复检未通过：在原文档更新 `review 结果/未通过原因`，不新建文档
