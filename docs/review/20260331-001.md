## Findings

### P0 - Critical

1. **并发安全问题：Stop() 方法死锁风险**

- 影响/场景：当 `Start()` 未被调用或 goroutine 未启动时调用 `Stop()`，会导致永久阻塞
- 定位：[`btc_dashboard_monitor_task.go:L50-L52`](../../internal/tasks/btc_dashboard_monitor_task.go#L50-L52)
- 建议：使用 select + default 避免阻塞：
```go
func (t *BtcDashboardMonitorTask) Stop() {
    select {
    case t.stop <- true:
    default:
    }
}
```

2. **并发安全问题：lastRunTime 数据竞争**

- 影响/场景：`lastRunTime` 在 goroutine 中写入，主线程可能读取，存在数据竞争
- 定位：[`btc_dashboard_monitor_task.go:L25`](../../internal/tasks/btc_dashboard_monitor_task.go#L25)
- 建议：使用 `sync.RWMutex` 或 `atomic.Value` 保护该字段

### P1 - High

1. **服务提前初始化问题**

- 影响/场景：即使 `BtcDashboardMonitor.IntervalSeconds = 0`（未启用），服务仍会被创建，浪费资源
- 定位：[`init.go:L32-L46`](../../internal/tasks/init.go#L32-L46)
- 建议：将服务创建移到条件判断内部：
```go
if cfg.BtcDashboardMonitor.IntervalSeconds > 0 {
    btcDashboardService := service.NewBtcDashboardService(...)
    // ...
}
```

2. **ahr999 时间戳逻辑存疑**

- 影响/场景：使用 K 线开盘时间替代当前时间计算 ahr999，可能导致结果与预期不符
- 定位：[`btc_dashboard_service.go:L69-L71`](../../internal/service/btc_dashboard_service.go#L69-L71)
- 建议：确认业务逻辑是否正确，添加注释说明为何使用 K 线时间而非当前时间

### P2 - Medium

1. **魔数缺乏说明**

- 影响/场景：ahr999 公式系数 `5.84` 和 `17.01` 缺少来源说明，难以维护
- 定位：[`btc_dashboard_service.go:L74`](../../internal/service/btc_dashboard_service.go#L74)
- 建议：定义为常量并添加注释：
```go
const (
    ahr999CoefficientA = 5.84
    ahr999CoefficientB = 17.01
)
```

2. **状态阈值硬编码**

- 影响/场景：ahr999 和 WMA200 的阈值判断硬编码，难以维护和调整
- 定位：[`btc_dashboard_service.go:L104-L129`](../../internal/service/btc_dashboard_service.go#L104-L129)
- 建议：抽取为配置或常量，便于后续调整

3. **减半计算精度问题**

- 影响/场景：假设每 10 分钟出一个块，实际出块时间波动（约 9-11 分钟）
- 定位：[`btc_dashboard_service.go:L87`](../../internal/service/btc_dashboard_service.go#L87)
- 建议：添加注释说明这是估算值，或考虑使用更精确的平均出块时间

### P3 - Low

- 建议：引入 mock 接口进行真正的单元测试

2. **错误变量未完全检查**
